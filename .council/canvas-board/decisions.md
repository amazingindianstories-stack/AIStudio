# Decision Log — Canvas Board

## D1 — Clone target: FigJam, not Figma design tool
**Decision**: Build FigJam-style whiteboard primitives (shapes, text, stickies, frames, connectors, infinite canvas). Do not attempt vector pen/bezier tool, boolean path ops, components/variants, auto-layout, constraints, or dev-mode/code export.
**Alternative rejected**: Literal "exact Figma functionality" — rejected because it is an infeasible multi-year scope for a single build, and the user's own reference screenshot is FigJam, not the Figma design tool.
**Asked user**: Yes — this changed what gets built entirely and no defensible default existed. User confirmed FigJam-style whiteboard.

## D2 — Asset library embedded as a canvas-side panel, drag/click to place
**Decision**: Reuse existing History/Favorites data and `/api/media/[...path]` proxy; no new asset-fetch or signing logic. Panel lives inside the canvas tab.
**Alternative rejected**: Read-only asset browsing with manual upload/paste only — rejected per user's explicit choice; drag/click placement is core to why this feature is being built (spatial storyboard assembly from existing generations).
**Asked user**: Yes, batched with D1.

## D3 — Boards persist per-project to Postgres + S3
**Decision**: New `boards` (or similar) table following the `projects`/`folders`/`generations` convention in `src/lib/schema.ts`; board JSON content stored per architect's judgment (S3 pointer vs jsonb column) based on existing schema/storage conventions.
**Alternative rejected**: Client-side-only (localStorage) persistence — rejected per user's explicit choice; boards need to survive reload and be usable across sessions like other generations.
**Asked user**: Yes, batched with D1.

## D4 — Single-user editing for v1, no real-time multiplayer
**Decision**: No CRDT sync layer, no live cursors/presence in this build. Explicitly deferred as a separate future project.
**Alternative rejected**: Real-time multiplayer from day one — rejected as disproportionate new infrastructure (websocket/CRDT server) for a v1; user confirmed single-user is acceptable to start.
**Asked user**: Yes, batched with D1.

## D5 — Stage 1 gate: design.md + ui-spec.md approved as-is, one reconciliation
**Decision**: Approved without a rework cycle. Both docs independently converged on the same architecture (top-level full-screen view via a new `Sidebar.tsx` rail icon, not a 4th `HistoryPanel` tab; separate `canvas-store.ts`; jsonb persistence; REST `[id]` blob route + op-switch metadata route). All 11 acceptance criteria are mapped to concrete mechanisms in both docs. Custom DOM/SVG rendering (rejecting tldraw for its commercial watermark license and store-format lock-in) is sound and the two hard correctness points (frame-membership delta propagation, connector endpoints stored as `{nodeId, anchor}` never coordinates) are concrete and unit-testable.
**Reconciliation**: `design.md`'s file plan names the rail icon `LayoutDashboard`; `ui-spec.md` (the authoritative visual contract, chosen closer to render time) specifies lucide `Shapes`. Resolved in favor of `ui-spec.md`: implementer uses `Shapes`. Not worth a rework round — a one-line icon choice, not a structural disagreement.
**Reservations logged, not blocking**: (a) the tldraw-watermark claim should be spot-checked against current tldraw pricing/license terms before fully closing the door on it for a *future* phase, though the independent control/integration argument (owning the jsonb node model, native-DnD asset drop) stands regardless; (b) orphan boards on project delete are explicitly deferred cleanup, acceptable for v1 per design's own Risks section.
**Asked user**: No — this is a routine design-quality gate, not a scope or requirement conflict.

## D6 — Stage 2 post-build fixes (applied directly, not deferred to review)
**Decision**: Both implementers independently flagged the same three defects during their own verification; since they were already diagnosed with exact file/line and an unambiguous fix, I patched them myself immediately after both implementers reported done, rather than waiting for Stage 3 review to "rediscover" them:
1. `src/lib/canvas/geometry.ts` `connectorPath` drew a straight `M...L` line; `ui-spec.md` §7 requires curved/bezier connectors by default. Changed to a quadratic bezier (`M...Q...`) with a perpendicular control-point offset capped at 60px. Verified against the existing `connectorPath` tests, which assert stability/non-crashing, not an exact path string — no test changes needed.
2. `src/lib/canvas-store.ts` `addConnector` defaulted new connector `stroke` to `#000000` — invisible against the `ink-900` (`#070708`) canvas background. Changed to `rgba(255,255,255,0.7)`, matching `ui-spec.md` §7's specified connector stroke exactly.
3. `src/lib/canvas-store.ts` `MIN_ZOOM`/`MAX_ZOOM` were `0.05`/`8`, inconsistent with `ui-spec.md` §2's pinned 10%–400% bounds (the UI implementer had already independently enforced 10%–400% at the wheel-zoom/zoom-control layer, but the store's own `zoomToFit()` action used the wider range). Tightened the constants to `0.1`/`4` so every zoom path — wheel, controls, and `zoomToFit` — agrees.
**Alternative rejected**: Sending these back through a formal implementer round-trip — rejected as wasted cycles for three single-line, unambiguous, already-diagnosed fixes with no design ambiguity.
**Verification**: re-ran the full 82-test suite (still 82/82) and `npm run build` (clean) after the edits; `git status` confirms only the files in `design.md`'s file plan were touched.
**Asked user**: No — mechanical bug fixes matching the already-approved spec, not a scope decision.

## D7 — Stage 3 review adjudication (code-reviewer + security-reviewer; ui-designer Mode 2 pending)
**Fixed directly** (small, well-diagnosed, verified with the full test suite + `npm run build` after each round):
- CRITICAL: frame membership (`parentId`) was never actually assigned on node drop — `computeFrameMembership` only drove the drop-highlight visual, never wrote `parentId`. Fixed in `canvas-store.ts`'s `moveSelectionBy`: after `moveNodesBy`, recompute and write `parentId` for dragged non-frame nodes against the resulting frame set.
- MAJOR: an edit made during an in-flight autosave PUT could be silently dropped (the success handler unconditionally overwrote `saveStatus:"dirty"` back to `"saved"`, causing the mid-flight edit's own retry timer to no-op). Fixed by only marking `"saved"` if `history.present` is still reference-identical to what was actually sent.
- MAJOR: rapid board-switching could apply a stale board's GET response after a newer load had already started (no ordering guard). Fixed with a `loadGeneration` monotonic counter in `canvas-store.ts`; a load whose generation is stale by the time it resolves is a no-op.
- MAJOR: dragging the opacity slider committed a history snapshot on every `onChange` tick, flooding and evicting the 50-entry undo cap within one drag. Fixed by passing `{coalesce:true}` to `updateSelectedStyle`'s `mutateGraph` call, matching move/resize's existing gesture-coalescing.
- MAJOR: the "Line/Arrow" shape (armed via the Shapes popover) could not actually be created — connectors could only start/end on a node, so dragging on empty canvas did nothing. Fixed in `CanvasSurface.tsx`: connector drags may now start and/or end on a free `{x,y}` world point, matching design.md's "free-standing arrows are just connectors with free endpoints" model (the store's `addConnector`/`Endpoint` type already supported this; only the pointer-handling wiring was missing it).
- MINOR: `zoomToFit`/click-to-center used a hardcoded 1600×900 viewport guess instead of the real container size. Fixed with a `setViewportSize()` store action fed by `CanvasSurface`'s existing `ResizeObserver`.
- MINOR: `saveBoardData` reported `{ok:true}` even when the target board no longer existed (e.g. deleted from another tab), masking a lost autosave. Fixed by using Drizzle's `.returning()` and surfacing 404 when zero rows matched.
- Dead ternary in `StyleInspector.tsx`'s stroke `onChange` — removed (quality only, not a behavior bug).
- **Security (HIGH)**: the canvas image-upload path (and the pre-existing `saveAssetImage`/asset-upload path, which shares the same helper) accepted `image/svg+xml` data URLs with no MIME allowlist, served back same-origin via `/api/media` with no `nosniff` — a stored-XSS vector (SVG can embed `<script>`). Fixed at the choke point: `storage.ts`'s `splitDataUrl` now allowlists JPEG/PNG/WebP/GIF only and throws on anything else (closes the gap for canvas AND the pre-existing asset/reference-image paths); added `X-Content-Type-Options: nosniff` to the media proxy response as defense in depth.
- **Security (MEDIUM)**: `ImageNode.src` wasn't constrained to the app's own media proxy, so a `PUT` could persist an arbitrary external URL that every future viewer's browser would auto-fetch (IP/UA/referrer leak to attacker infra). Fixed in `validateCanvasState`: image nodes now require `src` to start with `/api/media/`, dropped otherwise (same treatment already given to `data:`-prefixed values).
- **Security (LOW/MEDIUM)**: no size caps existed on the autosaved board JSON blob or on canvas image uploads. Fixed: a 2MB cap on the `PUT` body (413 if exceeded) and an 8MB byte cap on `saveCanvasAsset` (mirroring the existing `saveAvatarImage` 3MB pattern).
**Deferred, not fixed** (reviewer flagged as "debatable"/matches real product convention): plain mouse-wheel scroll pans rather than zooms (ctrl/cmd+scroll zooms) — this is FigJam's actual default behavior, not a bug; noted for the user in case they'd prefer plain-scroll-to-zoom instead.
**Asked user**: No — all fixes are either already-approved-spec conformance or uncontested security hardening; none change product scope.

## D8 — Stage 3 review adjudication (ui-designer Mode 2, live rendered UI)
The resumed live-UI review (Playwright against a real running dev server, logged in as the seeded admin) verified 9 of 11 acceptance criteria working end-to-end, including AC #9 (persistence-on-reload — the hardest one), and found one CRITICAL bug plus six MINOR ones.
**Fixed directly:**
- CRITICAL: `ConnectorLayer.tsx`'s overlay `<svg>` was `width={0} height={0}` — per the SVG spec this disables rendering of the element entirely, `overflow: visible` does not override it. Every connector, arrowhead, the live connector-drag rubber-band, and the marquee-select rectangle were silently painting nothing despite correct DOM/paths (the D6 curve/color fixes were correct but invisible). Fixed by using `width={1} height={1}` (any nonzero box keeps rendering enabled; children still draw outside it via `overflow: visible`, unchanged from the original intent). Verified by a fast, narrowly-scoped follow-up visual check (not a full re-review).
- MINOR: `ColorSwatchButton`'s inline styles mixed the `background` shorthand with a `backgroundSize` longhand in the same style object, triggering a persistent React dev-overlay warning. Fixed by switching to explicit `backgroundImage`/`backgroundColor` longhands.
- MINOR: the Shapes-popover trigger button had no accessible name (`aria-label` was on an inner `<span>`, not on the actual `<button>` that `Dropdown.tsx` renders around it). Fixed by passing `label="Shapes (R)"` to the `Dropdown`.
- MINOR: the "Add image" control was a `<label>` wrapping a `display:none` file input — neither element is keyboard-focusable, breaking the documented Tab path. Fixed by converting it to a real `<button>` that triggers a ref'd hidden input via `.click()`.
- MINOR: the Sidebar rail buttons (Image/Video/Board) exposed their label only via a hover-only motion tooltip, no `aria-label`/`title` on the button itself. Fixed once in the single shared `renderItem` button, covering all three (Board included, since it reuses the same pre-existing renderer).
- MINOR: the delete-board confirm dialog's Cancel button had no visible focus-ring and focus didn't reliably land on it (the dialog opens from a `Dropdown` row menu whose own `closeAndRestore` schedules a competing rAF-delayed refocus of its trigger, which could win the race against a synchronous `.focus()` call). Fixed with a double-rAF defer (so this dialog's focus always applies after the menu's restore-focus) plus `focus-visible:ring-2` on both dialog buttons.
**Deferred, not fixed** (would require extending the approved data model, not a bug fix): Bold toggle for text/sticky, text-align control for sticky (both listed in `ui-spec.md`'s table but `TextNode`/`StickyNode` in `design.md`'s approved data model have no backing field for them — `align` only exists on `TextNode`), and a 3-state (none/→/↔) arrowhead toggle (`Connector.kind` is a 2-state `"line"|"arrow"` per the approved data model). Also deferred: `FrameNode` on-canvas label low-contrast/positioning polish (cosmetic, non-blocking).
**Environment-only issues noted, not feature bugs**: the review agent had to re-seed/reset the local admin password hash and hit a stale `.next` cache — both sandbox artifacts, not shipped code. It also found `/api/media/*` returning 500s in its sandbox due to invalid S3 credentials in that environment — pre-existing shared infra, unrelated to this feature, not fixed here.
**Verification**: full 82-test suite still green and `npm run build` clean after every fix in this round; a fast, narrowly-scoped follow-up visual check (not a full re-review) re-confirmed the connector/marquee now render, re-route live on node move, and that the free-standing line/arrow tool (a related gap from D7) now works.
**Asked user**: No — all fixes are conformance to the already-approved spec/design or accessibility hardening; nothing changes product scope.
