# Recon Summary — Canvas Board

## Frontend

- `src/app/page.tsx`: 3-column shell — `TopBar`, `Sidebar` (44px left rail, currently only has AI Image/Video mode switch), `ConversationPanel` (feed) + `PromptComposer`, right collapsible panel hosting `HistoryPanel`. `DetailModal` at root.
- The Project/History/Favourites tab switcher actually lives **inside `src/components/HistoryPanel.tsx`** (~line 137-151), not in `Sidebar.tsx`. `TabBtn` local component; body switches on `rightTab` (~line 197) to render `ProjectPanel` / favourites grid / history feed.
- `src/lib/store.ts` (945 lines): single flat Zustand store, no slices, no `zustand/middleware` (no persist/immer). Relevant existing fields: `rightTab: "project"|"history"|"favorites"`, `activeProjectId`, `activeFolderId`, `selectedIds` (grid multi-select only — **no zoom/pan/viewport/canvas-selection precedent exists anywhere**). UI-state persistence is a hand-rolled `useStore.subscribe` diff-and-write to localStorage, debounced 400ms for the prompt field, synchronous for others (lines ~829-945).
- Components (`src/components/`, flat, 14 files): `TopBar`, `Sidebar`, `ConversationPanel`, `PromptComposer` (has native HTML5 DnD for ref images), `HistoryPanel` (owns tab bar + grids), `ProjectPanel` (folder rail + masonry grid, native DnD drag-to-folder), `AssetLibrary` (reusable reference asset CRUD, described in codebase comments as "dormant in the UI"), `MediaCard` (shared thumbnail card for image/video, used by HistoryPanel + ProjectPanel), `DetailModal`, `Dropdown`, `MentionTextarea`, `AccountSettings`, `AdminDashboard`.
- **Drag-and-drop**: native HTML5 DnD only (`draggable`, `onDragStart`/`onDragOver`/`onDrop`, `DataTransfer`), no dnd-kit/react-dnd anywhere. Pattern: `dataTransfer.setData("text/itemId", id)` on drag source, drop target reads it and calls a store action. Reuse this exact pattern for asset→canvas placement.
- **No canvas/whiteboard library installed** — no konva/react-konva, fabric, tldraw, reactflow, dnd-kit. Present but not canvas-specific: `framer-motion@^11`, `zustand@^5` (no middleware), `lucide-react`, `recharts` (admin only). **Net-new dependency decision is the architect's to make and justify.**
- Thumbnails: components consume a pre-resolved `item.url` (already `/api/media/<key>` form) directly in `<img>`/`<video>` — no client-side URL building needed for canvas image nodes either.

## Backend

- `src/lib/schema.ts`: tables `users, projects, folders, generations, assets, pricing, activityLogs`. No Drizzle `.references()`/`relations()` anywhere — all FKs are bare `uuid` columns with manual cascade cleanup in `-db.ts` files. Timestamps: `bigint("...", {mode:"number"}).notNull()` populated with `Date.now()`. **`jsonb` is already an established pattern** — `generations.referenceImages: jsonb().$type<string[]>()`, `assets.images: jsonb().$type<string[]>().notNull().default([])`, `activityLogs.detail: jsonb()`. → A `canvas_boards.data: jsonb("data").$type<CanvasState>().notNull()` column matches convention exactly; no need for an S3-pointer indirection for board JSON.
- ID convention is mixed: `projects`/`folders` use `.defaultRandom()` (DB-generated); `generations`/`assets` use app-supplied `crypto.randomUUID()` (needed pre-insert to key storage objects). Canvas Board doesn't need to key storage objects by board id before insert, so DB-generated id is fine, OR app-supplied for consistency with needing the id client-side immediately on create — architect's call.
- Data access layer convention (`src/lib/*-db.ts`): one file per table/domain. `projects-db.ts` re-reads and returns the *full* list after every mutation (`readProjects()`) — explicitly flagged as NOT a good fit for a board document (should return just the single board, not "all boards"). `assets-db.ts` shows an `onConflictDoUpdate` upsert pattern — good fit for periodic full-document board saves.
- API route convention: flat collection routes, no `[id]` dynamic segments used for projects (`src/app/api/projects/route.ts`: `GET` + a single `POST` switched on `op` string, always returns full updated list). Auth via `getSession()` inline-null-check or `requireUser()`/`requireAdmin()` (throws, catch → 401/403) — both patterns coexist. Body parsing always `await req.json().catch(() => ({}))`.
- `src/lib/storage.ts` / `src/lib/save-media.ts`: `uploadBuffer/uploadBase64/uploadFromUrl` all return `/api/media/${key}` (never raw S3 URL); `save-media.ts` wraps these with app-level key-naming (`generations/${id}.${ext}`, `assets/${uuid}.${ext}`, etc.). A canvas-attached upload (e.g. pasted/uploaded image not from the existing library) should get its own `saveCanvasAsset(dataUrl, boardId)`-style wrapper following this shape.
- **No existing server-write debounce/autosave utility.** The only debounce precedent is the client-only localStorage one in `store.ts` (400ms setTimeout on store.subscribe diff). A debounced POST-to-API autosave for board content is new territory — architect should design it explicitly (debounce window, conflict handling on tab-switch/unmount flush, failure/retry UX).
